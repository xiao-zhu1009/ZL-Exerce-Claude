# 用户端：饮食中心与饮食记录整合说明

本文档将原「饮食中心」「饮食记录」两个入口的产品目标、页面职责与接口依赖合并为一份说明，并给出**统一为单一饮食中心**后的信息架构与落地建议，便于评审与排期。

---

## 1. 背景与目标

### 1.1 问题

- 侧栏存在 **「饮食中心」** 与 **「饮食记录」** 两个菜单，用户需在相近功能间切换路由，认知成本高。
- 两模块 UI 形态相近（卡片、Tab、表格/图表），拆成两个页面容易造成「像两套功能」的割裂感。

### 1.2 目标

- **一个用户心智模型**：饮食相关能力（看内容 → 查食物 → 记饮食 → 看趋势/计划）集中在一个 **「饮食中心」** 内完成。
- **一个侧栏入口**：仅保留「饮食中心」，进入后通过 **Tab** 切换子模块。
- **兼容旧链接**：历史地址 `/user/diet/record` 通过重定向或查询参数进入同一页面中的「饮食记录」Tab，避免收藏夹失效。

---

## 2. 功能范围（合并后的能力清单）

| 能力 | 说明 | 原归属 |
|------|------|--------|
| 推荐餐食 | 饮食文章列表、分类/关键词筛选、分页、进入详情 | 饮食中心 |
| 食物库 | 食物搜索、分类筛选、展示每 100g/ml 营养素；支持「添加到今日记录」快捷录入 | 饮食中心 |
| 饮食记录 | 按日查看各餐次记录、增删改；近期趋势与 ECharts 图表、区间汇总 | 饮食记录 |
| 我的饮食计划 | 教练下发/系统计划列表与详情（若已对接） | 饮食中心 |

---

## 3. 目标信息架构（整合后）

### 3.1 侧栏

- 仅一项：**饮食中心** → `/user/diet`

### 3.2 饮食中心页内 Tab（建议顺序）

1. **推荐餐食** — 内容浏览与导流。
2. **食物库** — 查询参考，并可一键写入当日记录。
3. **饮食记录** — 原「饮食记录」页中的「每日详情 + 近期趋势」整体迁入本 Tab（内部可保留二级 Tab：每日详情 / 近期趋势）。
4. **我的饮食计划** — 与教练端下发的计划只读展示（与现有 `dietPlan` 能力对齐）。

### 3.3 顶栏与按钮

- 去掉独立路由跳转的「我的饮食记录」大按钮；若需快捷操作，改为 **切换到「饮食记录」Tab**（不离开当前页）。

### 3.4 路由约定（建议）

| 路径 | 行为 |
|------|------|
| `/user/diet` | 饮食中心主页；可用查询参数 `tab` 指定默认 Tab，如 `?tab=record` |
| `/user/diet/articles/:id` | 文章详情（保持独立子页，便于分享链接） |
| `/user/diet/record` | **重定向**至 `/user/diet?tab=record`（或等价命名，全项目统一即可） |

---

## 4. 与后端接口的对应关系

前端封装见 `src/api/diet.js`、`src/api/dietPlan.js`（以仓库内实际导出为准）。

### 4.1 文章与食物（饮食中心）

- 文章列表/详情：`GET /diet/articles`、`GET /diet/articles/{id}`
- 食物搜索：`GET /diet/foods/search`
- 快捷添加记录：`POST /diet/records`（与饮食记录同源）

### 4.2 饮食记录

- 某日记录：`GET /diet/records?record_date=`
- 区间汇总（趋势图）：`GET /diet/records/range`
- 新增/修改/删除：`POST /diet/records`、`PUT /diet/records/{id}`、`DELETE /diet/records/{id}`

### 4.3 饮食计划

- 用户端列表/详情：`GET /diet-plans`、`GET /diet-plans/{id}`（见 `dietPlan.js`）
