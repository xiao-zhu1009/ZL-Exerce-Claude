# 用户端数据看板设计

> 路由：`/user/dashboard`（用户登录后默认落地页之一）  
> 页面：`src/views/user/Dashboard/index.vue`  
> 本文档汇总**现状、推荐可视化、后端接口映射与落地注意点**，便于前后端对齐实现。

---

## 1. 设计目标

- 用少量**主图表 + KPI 卡片**，让用户一眼看出：**练没练、练得均不均衡、吃与动的关系、体重/体态是否在朝目标变化**。
- 数据以**真实接口**为准，避免长期依赖 `src/api/user.js` 中的 mock。
- 同一屏建议 **3～4 个主图**；更多分析可放到二级 Tab 或「展开更多」，避免信息过载。

---

## 2. 当前实现与差距

| 项目 | 说明 |
|------|------|
| 概览与部分图表数据 | `getDashboard()` 仍为 `mockResolve`，未走 `utils/request` |
| 训练柱状图 | `initWorkoutChart` 内为写死的演示数据，与真实训练无关 |
| 体重趋势 | 依赖「历次体重记录」；后端存在 `body_records` 表，但 **API 尚未暴露**（仅有 `GET/PUT /user/body-stats` 当前快照） |
| 文案口径 | 若卡片写「本周」，需与统计区间一致：自然周（周一至周日）与「近 7 天滚动」含义不同 |

详细模块说明可与 `docs/01-用户端.md` 中「数据看板」一节对照阅读。

## 3. 推荐可视化清单（优先级）

### 3.1 建议优先（主看板）

1. **近 7 天 / 自然周训练量**（柱状图）  
   - 指标：每日**训练时长（分钟）**或**消耗热量（kcal）**。  
   - 价值：这周有没有练、哪几天空档、量是否稳定。

2. **训练类型分布**（饼图 / 环形图）  
   - 指标：近 30 天各 `workout_type` 占比。  
   - 价值：是否训练过于单一（如有氧过多、力量不足）。

3. **摄入 vs 消耗（热量平衡）**（双轴折线 / 分组柱 + 折线）  
   - 指标：按日的**饮食热量**与**运动消耗**（或日净热量 = 摄入 − 消耗）。  
   - 价值：与减脂/增肌目标结合时，直观看出热量缺口或盈余是否合理。

4. **KPI 卡片（大数字 + 可选环比）**  
   - 例如：本周训练次数、本周消耗总和、今日摄入、当前体重；可选「较上周 +/−」。  
   - 价值：快速扫一眼完成度与坚持情况。

### 3.2 有能力再上

5. **体重 / 体脂趋势**（折线图）  
   - 依赖**历史体测或体重记录**接口；仅有当前 `body-stats` 时只能展示单点或弱化空状态。

6. **宏量营养素趋势**（堆叠柱：蛋白 / 碳水 / 脂肪）  
   - 基于饮食区间汇总按天展示，适合与饮食目标联动。

7. **计划完成 / 打卡热力**（若有计划状态与日历数据）  
   - 强化「习惯」与长期坚持感。

## 4. 后端接口映射（FastApi，`/api` 前缀）

| 能力 | 方法 | 路径 | 用途 |
|------|------|------|------|
| 近 N 天每日训练时长与消耗 | GET | `/training/stats?days=` | 训练柱/折线、周消耗汇总 |
| 近 N 天训练类型分布 | GET | `/training/stats/type?days=` | 饼图 |
| 训练记录列表 | GET | `/training/records?start=&end=` 或 `days=` | 本周条数、自定义区间聚合 |
| 某天饮食记录 | GET | `/diet/records?record_date=`（不传则默认今天） | 今日摄入求和 |
| 日期区间每日营养汇总 | GET | `/diet/records/range?start=&end=` 或 `days=` | 摄入趋势、与训练对比 |
| 当前身体指标快照 | GET | `/user/body-stats` | 当前体重等 KPI |
| 体重历史（待开发） | — | 建议 `GET/POST /user/body-records` 等 | 体重折线图 |

前端已有封装可参考：`src/api/training.js`、`src/api/diet.js`；看板应逐步改为与上述接口一致，而非单一未实现的 `GET /user/dashboard` mock。

## 5. 实现策略（两档）

### 方案 A：前端多接口编排（推荐先做）

在 Dashboard 进入时 `Promise.all` 并行请求，在页面内组装卡片与图表：

- **本周训练次数**：`getWorkoutRecords` 传入本周 `start`/`end`，条数即次数（产品需确认是否「按天最多计 1 次」）。
- **本周消耗**：对 `getDailyStats(7)` 返回数组的 `calories` 求和（注意与「自然周」文案一致）。
- **今日摄入**：`getDietRecords()` 不传日期，对返回列表 `calories` 求和。
- **当前体重**：`GET /user/body-stats` 的 `weight`。
- **训练主图**：`getDailyStats(7)` 的 `date` + `duration` 或 `calories`。
- **类型分布**：`getTypeStats(30)`。
- **摄入趋势 / 对比**：`getDietRangeSummary` 与 `getDailyStats` 按日期对齐后绘图。

### 方案 B：后端聚合 `GET /user/dashboard`（可选）

一次返回 overview + 序列数据，减少请求、统一口径；仍需补全 **体重历史** 等缺失能力时才有完整体验。

## 6. 前端实现注意点

1. **响应结构**：`src/utils/request.js` 拦截器返回 `res.data`，与各页面统一解析 `code` / `data` / `msg`，避免有数据却不渲染。
2. **空数据**：无训练、无饮食、无体重记录时，使用 `el-empty` 或 ECharts 空配置，避免报错。
3. **周口径**：自然周需前端算本周一、周日日期，再请求 `records`；`stats?days=7` 为「含今天在内的连续 7 天」，卡片文案建议与实现一致（「近 7 天」或「本周」二选一）。
4. **图表销毁**：路由切换或重绘时按需 `dispose` ECharts 实例，防止内存泄漏（可参考 `TrainingChart.vue`）。
5. **移除 mock**：`src/api/user.js` 中与看板、个人资料相关的 TODO，在对接真实接口后删除 `mockResolve` 路径，避免环境与演示数据混淆。
